# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

FCFC '26 is a private World Cup 2026 prediction game (in Turkish, slang "aga") for a fixed group of 6 friends. It is a single-page Next.js App Router PWA backed by Firebase Firestore with realtime sync. There is no signup — players are hardcoded and "log in" with a 4-character code.

## Commands

```bash
npm run dev      # dev server at http://localhost:3000
npm run build    # production build
npm start        # serve production build
npm run lint     # eslint (eslint-config-next)

# One-time: seed the 72-match WC 2026 group stage into Firestore
# (after filling NEXT_PUBLIC_FIREBASE_* in .env.local)
node --env-file=.env.local scripts/seed.mjs
```

There is no test suite. `@/*` resolves to the repo root (tsconfig paths).

## Backend setup (Firebase)

The app needs a Firebase project (free Spark plan, no card). `.env.local` holds the `NEXT_PUBLIC_FIREBASE_*` web-app config (public client keys — access is gated by Firestore rules, not by hiding them) plus `ARBITER_CODE` (used only by the `scripts/`; the app itself never sees the code). One-time console setup before the rules will work:
1. **Authentication → Sign-in method → enable Anonymous.** Every client (and the poller) signs in anonymously so the rules can tell callers apart — there are still no accounts.
2. **Firestore → create `config/arbiter` = `{ code: "<your code>" }`.** This is the arbiter code; it's never client-readable, so it stays out of the bundle.
3. **Firestore → Rules → publish `firestore.rules`.**

Reads stay public; writes require auth + a valid shape, and only an arbiter can change results (see Arbiter mode). **Rollout order matters on a live deploy:** enable Anonymous Auth and create the config doc *before* publishing the rules, and deploy the app (which adds the anonymous sign-in) *before* the rules, or in-flight clients lose write access until they refresh. Then run the seed command above once.

## Architecture

**Single-screen, four-tab app.** `app/page.tsx` (`'use client'`) is the entire UI shell and owns nearly all state: the active tab (`next` | `upcoming` | `past` | `table`), the current user, the players/matches/bets caches, `isArbiter`, and `nowMs` (a clock ticked every 60s). Tabs: **Up Next** (bettable `MatchCard`s), **Upcoming** (future fixtures, not yet bettable), **Past** (finished cards with outcome coloring), **Table** (`Leaderboard`). Almost everything is a client component — Firestore realtime listeners and localStorage require it.

**Betting window & tab partitioning** (`lib/data.ts` helpers): a match's kickoff is `kickoffMs(m)` (parses the Turkey-time `date`/`time` strings → UTC ms). Non-finished matches split by kickoff vs `nowMs + BET_WINDOW_MS` (48h):
- *Up Next* (`next`) = `kickoff <= now + 48h` — the bettable window. A card is `locked` (betting closed, shows "CLOSED"/"LOCK") once `nowMs >= kickoff` but still awaiting a result.
- *Upcoming* = `kickoff > now + 48h` — future fixtures, passed `notYetOpen` (betting not open, shows "SOON"; arbiter entry suppressed via `showArbiter`).
- *Past* = `FINISHED` only.

So a match flows Upcoming → Up Next → (locked) → Past, the last step driven by the arbiter finalizing it. The arbiter can also reverse it: the "Reopen Match" button on a finished card calls `handleReopenResult`, which sets `status: 'UPCOMING'` and `deleteField()`s both scores, dropping the match back into the flow (bets are kept).

**Data flow is Firestore-direct, no API layer.** `lib/firebase.ts` initializes one browser app (HMR-safe via `getApps()`) and exports `db`. Components import `db` and the modular `firebase/firestore` functions directly. `page.tsx` opens two `onSnapshot` listeners — on the `matches` and `bets` collections — which both backfill and stream live updates (no separate initial fetch). Writes optimistically update local state, then `setDoc`/`updateDoc`; the snapshot listener reconciles with what other clients write. `isLoading` clears on the first `matches` snapshot.

**Firestore collections:**
- `matches/{id}` — doc id is the stringified numeric `id`. Fields: `id, home, away, date, time, stadium, status, result_home?, result_away?` (the `Match` interface in `lib/data.ts`). Times are stored in Turkey time (the seed converts from ET). Sorted client-side by `id`. Seeded by `scripts/seed.mjs`.
- `bets/{user_id}_{match_id}` — **the doc id IS the client bet key** (`` `${user_id}_${match_id}` ``). Fields: `user_id, match_id, pick` (`'HOME' | 'DRAW' | 'AWAY'`), `locked` (bool). Outcome-only — no scoreline. `handlePick` upserts the pick (`locked: false`); `handleLockIn` flips `locked: true`. Legacy score-based bet docs (no `pick`) are ignored on read.
- `players/{uuid}` — self-registered roster. Fields: `id, name, avatar, pin` (the `Player` interface). `id` is a `crypto.randomUUID()`.
- `brackets/{user_id}` — a player's locked "Build Your Bracket" prediction (`BracketPrediction`: `user_id, picks, locked, updatedAt?`). `picks` maps each of the 31 knockout match ids (R32→Final, no third-place game) → the team name they predict to advance. Written **once and locked** — the rules reject any later update, so it's permanent. Drafts live only in `localStorage['pitch_bracket_draft_{user_id}']` until locked (the collection only ever holds finished, locked brackets).

**Players self-register; the roster lives in Firestore, not in code.** `page.tsx` streams the `players` collection live. The user switcher renders it plus a "+ New" button that opens `RegisterModal` (username + emoji avatar — `AVATARS` quick-picks or any typed emoji — + 4-char PIN → writes a `players` doc and logs in). Clicking an existing player opens `AuthModal`, which checks the typed code against `player.pin`. The logged-in player id persists to `localStorage['pitch_club_user']`. PINs are plaintext and world-readable — this is identity, not security; fine for a private pool. (The Firestore rules require an anonymous-auth session and a valid doc shape to write a profile, but they don't tie a player id to a uid, so the PIN is still the only thing guarding a profile switch — by design.)

**Arbiter mode** (the `⚖` button in the header) is gated by `ArbiterModal`. The code is no longer in the client: the modal proves knowledge of it by writing `arbiters/{uid}` with the entered code, which the rules validate server-side against the closed `config/arbiter` doc — a wrong code is rejected (permission-denied) and surfaced as "Wrong code". On success `isArbiter` flips on and a `localStorage['pitch_arbiter']` flag persists it across reloads (the anonymous uid and the `arbiters/{uid}` doc persist per device too, so authorization survives). It swaps every `MatchCard` into result-entry mode (writes `status: 'FINISHED'` + scores to `matches` — allowed only because `arbiters/{uid}` exists); finished cards also get a "Reopen Match" button to de-finalize. Clicking `⚖` again calls `handleExitArbiter`, which deletes `arbiters/{uid}` (self-delete) so the device loses result-writing rights, not just the UI. The poller and the `scripts/` register as a transient arbiter via `scripts/connect.mjs` (anon sign-in + `arbiters/{uid}` from `ARBITER_CODE`) and clean it up after each run.

**Live results poller** (`scripts/poll.mjs`, run every ~5 min by `.github/workflows/poll-results.yml`). Pulls the *unofficial* ESPN World Cup scoreboard (`site.api.espn.com/.../soccer/fifa.world/scoreboard`, no key) and writes live scores, the running `minute`, goalscorers/red cards (`events: MatchEvent[]`), and `status: 'LIVE'`/`'FINISHED'` straight into `matches` — the app's existing `onSnapshot` streams them to clients, so a `LIVE` match shows up in Up Next (locked, betting closed) with a live score + scorer list. It writes via the client SDK against the open rules (same `NEXT_PUBLIC_FIREBASE_*` config, no service account; CI reads them from repo secrets). Auto-finalizing is what triggers scoring; the **arbiter stays as the fallback/override** for anything the feed mis-scores or doesn't cover. ESPN team names are reconciled to the seed's names by a normalizer + `ALIASES` map in the script (e.g. `United States`→`USA`, `Bosnia-Herzegovina`→`Bosnia and Herzegovina`); unmapped teams are logged, not guessed. The feed is undocumented and can change shape/404 — the poller never throws on one bad match, and ESPN's scoreboard only returns the current day's games, so older `FINISHED` matches are left untouched. `poll.mjs` also takes an optional date arg (`node scripts/poll.mjs 20260611` or a range) to backfill past games — the `workflow_dispatch` has a `dates` input for running it from the Actions tab.

**Reliable poller cadence.** GitHub's `schedule` is best-effort (drops `*/5` runs under load), so the cron is only a fallback; the guaranteed cadence comes from an external scheduler (cron-job.org) POSTing to the `workflow_dispatch` API every ~10 min with a fine-grained/classic PAT (Actions: read+write). See the workflow header.

**Recent form + Groups tab.** `scripts/form.mjs` (run by `.github/workflows/form.yml`, a few times a day) pulls each nation's last-5 results across all competitions from ESPN's `all/teams/{id}/schedule` and writes `teams/{name}` docs (`TeamForm` = `{ name, form: FormMatch[] }`, most-recent first). The app streams `teams`; each `MatchCard` shows a W/D/L form strip per side (green/yellow/red, newest rightmost) that taps open to both sides' last-5 scorelines. The **Groups** tab (`components/Groups.tsx`) is fully self-contained: `GROUPS` in `lib/data.ts` is the static 12-group draw, and `computeStandings`/`groupMatches` derive each table and its completed scorelines from the `matches` already in state (points→GD→GF tiebreak; no external standings call). The `teams` collection is public-read, arbiter-write — **publishing the updated `firestore.rules` is required** before the form job can write.

**Starting XIs.** The "Line-ups (XI)" button on a ticket opens `LineupModal`, which fetches lineups *client-side* from ESPN (`lib/espn.ts`: resolve the event from the scoreboard by team+date, then read `summary?event=`'s `rosters` → formation + starters + subs). Both ESPN endpoints send `Access-Control-Allow-Origin: *`, so there's no server/Firestore involvement at all — nothing to store, no rules. The modal has a **Pitch/List toggle**: `Pitch.tsx` draws both XIs on an SVG pitch (home bottom, away top), positioning players from the formation string (starters arrive in formation order, GK first) with an L/R nudge derived from each position abbreviation. Lineups are only published ~1h before kickoff; before that the modal shows "not published yet". ESPN names are reconciled to ours via the same alias/normalizer approach as the scripts.

**Betting UX (`MatchCard`).** Outcome-only: the player taps the home flag, away flag, or center **Draw** chip to set `pick`; the selection is ringed gold. The pick is editable until **Lock In** (sets `locked`) or kickoff (auto-locks via the time-based `locked` prop — `canPick` goes false). Once a bet is committed (`locked` || kicked off || finished), the card shows **See Bets**, which lists other players whose bets are *also* committed and how they picked (gated so you must commit before you can peek). On finished cards the pick ring and the See Bets entries turn green/red by outcome.

**Scoring / bet outcome.** `outcomeOf(m)` is the scored result of a match: for a knockout it's `m.advance` (so a tie settled in ET/penalties scores the right picker), otherwise `resultOutcome(rh, ra)` reduces the scoreline to `HOME`/`AWAY`/`DRAW`. `betOutcome(pick, m)` returns `win` (pick matches `outcomeOf`), `loss`, or `none`. Correct pick = **1 point**, no exact-score tier. `Leaderboard.tsx` sums wins over finished matches (1 pt each), sorted by points.

**Knockout bracket (matches 73–104).** The group seed (`scripts/seed.mjs`) is only the 72 group games; the 32 knockout fixtures (Round of 32 → Final) are a *separate* one-time seed, `scripts/seed-knockout.mjs`, that writes **only ids 73–104** so it can't clobber finished group docs (**never re-run `seed.mjs` once results exist** — it would wipe them; always `node --env-file=.env.local scripts/backup.mjs` first regardless). Knockout `Match`es carry `stage` (`'R32'|'R16'|'QF'|'SF'|'THIRD'|'FINAL'`; group matches have none), and once finished, `advance` (`'HOME'|'AWAY'` — who progressed) and optional `shootout` (`{ home, away, takers? }`). They seed with placeholder slot labels for `home`/`away` ("Winner A", "Runner-up B", "3rd C/D/F/G/H", "Winner M73", "Loser M101"); `teamsResolved(m)` (both names are `TEAM_ISO` keys) gates betting — an unresolved slot stays in **Upcoming** (not bettable) until filled. The **poller** (`scripts/poll.mjs`) fills the real qualifiers as each round resolves: a bracket-fill pass matches an ESPN fixture to a slot by kickoff time (within 2h) and overwrites `home`/`away` (adopting ESPN's home/away), and on a knockout final it writes `advance` (from ESPN's `winner` flag) + `shootout` (from `shootoutScore`/shootout-period details). In the UI (`MatchCard`) knockout cards drop the **Draw** chip (pick who advances — HOME/AWAY only), show a stage tag, render the penalty tally + takers, and the arbiter gets an *Advances* toggle + penalty `ScoreDial`s (shown when regulation is level) as the manual fallback. The Groups tab is unaffected (it filters by group membership).

**Build Your Bracket (`components/BracketGame.tsx`, in the Bracket tab below the live tree).** A prediction minigame: a logged-in player taps "Build Your Bracket" to enter build mode and clicks who advances at each of the 31 knockout ties (R32→Final; the third-place game is excluded). Each pick **propagates upward** — the team you send through fills a slot in the next round (`bracketSides(id, picks, byId)`: R32 nodes read the live fixture teams, every higher node reads the predicted winners of its two feeders via `BRACKET_FEEDERS`). Changing a lower pick auto-clears any now-stale higher pick (`prunePicks`, cascades low→high in one pass). A bracket completes at 31/31 and "Lock In" is **permanent** (a two-tap confirm; the rules forbid updating a locked doc). Drafts persist only in `localStorage` until locked. Locked brackets are listed for everyone ("ACC's Bracket") and tap to expand a read-only `PredTree`; the list is **sorted by `bracketAccuracy`** (`% Accurate` = correctly-predicted advancers over finished knockout ties, `null`/"—" until the first ties settle), colouring each pick green/red once its real tie is decided. The minigame is purely cosmetic — it does **not** feed the `Leaderboard`. Reuses the live tree's geometry (`H/TOTAL/NODE_W/CONN_W/Connector/STAGE_LABEL` are exported from `Bracket.tsx`). **Publishing the updated `firestore.rules` is required** (adds the `brackets` collection rules) before lock-in can write.

**Team flags** are rendered from country name via `TEAM_ISO` (`lib/data.ts`) → `https://flagcdn.com`. England/Scotland use special `gb-eng`/`gb-sct` codes; unmapped names fall back to a `?` swatch. `TEAM_ISO` keys must match the team names in `scripts/seed.mjs` exactly — both cover the same 48 WC-2026 teams (e.g. `Türkiye`, `Czechia`, `DR Congo`, `Bosnia and Herzegovina`).

## Design constraints (from `.cursorrules` — these are intentional, do not "modernize" away)

The aesthetic is deliberately anti-startup: "1970s Match Programme meets Swiss Editorial." Custom theme colors live in `tailwind.config.ts` — `pitch` (greens), `paper` (bone text), `gold`, `signal` (alert orange), `chalk` (borders). Fonts: `font-serif` = Fraunces (headers), `font-mono` = Chivo Mono (data), wired up as CSS variables in `app/layout.tsx`.

- No flat backgrounds — layer noise/grain.
- No blue buttons, drop shadows, or `rounded-full` "startup" buttons; use the gold/signal palette and tactile controls. Score entry uses the custom `ScoreDial`, never standard number inputs.
- Functional components, meaningful names (`matchData`, not `data`), Framer Motion for staggered/smooth motion.
